# Servlet下的Filter鉴权

## web.xml

1.  根据web.xml路由去查看各种操作（doGet与doPost）的鉴权：  
    首先去找 web.xml 文件，然后对着 web.xml 文件去找 `<url-pattern>` 标签，该标签对应着我们要去访问的URL路径，然后再去找他所对应的 `<servlet-name>` 这里面就是所对应的类代码
2.  Filter也同样可以在web.xml中去配置

## Filter

1.  注解：`@WebFilter`
2.  `doFilter`：Filter的生命周期：init -> doFilter -> destroy，所以主要审计`doFilter`方法对请求放行：chain.doFilter(request,response)
3.  搜接口实现：`implements Filter`

# Shiro鉴权

1.  搜配置Bea：搜索 `ShiroFilterFactoryBean`、`shiroFilter`  
    在这个类中进行路由鉴权的时候，找 `filterChainDefinitionMap.put("/getName","anon");` 这样的代码，只要后面是有`anon`这个参数的，就表示该路径不需要鉴权
2.  搜注解：`@RequiresPermissions`、`@RequiresRoles`、`@RequiresAuthentication`

# Spring下的路由鉴权

1.  配置文件：WEB-INF/classes目录下 springMVC.xml/springMVC-servlet.xml，进入配置文件之后，直接搜索`interceptor`
2.  查找拦截路径：`implements WebMvcConfigurer`、`addInterceptor`方法、Interceptor  
    搜 `implements WebMvcConfigurer`，找 `addInterceptors(InterceptorRegistry registry)` 方法，这里定义了拦截器拦截和排除的 URL 路径（`addPathPatterns` / `excludePathPatterns`）。
3.  拦截的具体实现：一般先搜`implements WebMvcConfigurer`，然后看 `preHandle` 方法。返回 `true` 放行，返回 `false` 拦截。
4.  注解：@PreAuthorize

# Jwt鉴权

依赖库：`io.jsonwebtoken` (`jjwt`), `java-jwt`, `nimbus-jose-jwt`

核心类：`JwtUtils`, `TokenFilter`, `JwtProvider`

高危方法：`parse()`, `decode()`, `parseClaimsJwt()` (注意没有 `s`)

安全方法：`verify()`, `parseClaimsJws()` (带有 `s`，代表校验 Signature)

配置文件：`jwt.secret`, `jwt.key`, `rsa.public-key`

| **漏洞类型** | **关键代码/配置特征** | **红队验证思路 (高层次)** |
| --- | --- | --- |
| **1. 仅解析不验签** | 使用了 `Jwts.parser().parse(token)`或自行 Base64 解码。 | 保持签名不变，直接篡改 Payload 中的 `userId`为 Admin，重放请求。 |
| **2. 硬编码/弱密钥** | 代码写死 `SECRET="123456"`<br>或使用超短字符串。 | 提取源码密钥，或抓取正常 Token 进行离线字典爆破，随后自行签发。 |
| **3.`alg:none`绕过** | 未强制白名单限制加密算法，被动信任客户端传入的 `alg`。 | 将 Header 篡改为 `alg:none`<br>，删除 Signature 部分，篡改 Payload。 |
| **4. 算法混淆降级** | RS256 降级为 HS256（非对称变对称）。 | 提取公钥，将其作为对称加密的 Secret，构造 `alg:HS256`并重新签名。 |
| **5. 注销失效/会话复用** | `logout`接口仅前端清 Token；无 Redis 黑名单机制拦截 `JTI`。 | 退出登录/修改密码/权限被降级后，继续使用旧的未过期 Token 访问核心接口。 |
| **6. Header 注入扩展** | 滥用 `kid`(密钥ID) 或 `jku`(密钥URL)，且未做过滤。 | 构造 `kid: "../../../dev/null"`<br>绕过验签，或引发 SQL 注入/SSRF。 |

# AOP切面与自定义注解

AOP切面与自定义注解

1.  自定义注解：全局搜索 `@interface`，寻找命名类似于 `@CheckLogin`、`@CheckAuth`、 `@Auth`、 `@RequireToken`、 `@IgnoreAuth`（常用于白名单放行）的注解。
2.  AOP切面：全局搜索 `@Aspect` 和 `@Around`、`@Before`。
3.  在 `@Aspect` 修饰的类中，看看切点（Pointcut）是不是绑定了 Controller 层的路径（如 `execution(* com.example.controller..*.*(..))`），或者绑定了前面找到的自定义注解。真正的鉴权逻辑就写在这个切面的方法里。

## URL解析差异绕过

### tomcat/springboot

tomcat 容器下，使用 `getRequestURL()` 与 `getRequestURI()` 两种方法可以进行绕过，如果采用 `getServletPath()` 来获取请求路径的话，使用 ; 来绕过就是失败的。

### 后端路径规范化

### /../解析

访问路由为：/login.html/../caigo

#### Tomcat 9.x

路径解析：Tomcat 默认会处理路径中的 ../ 进行规范化

```plain
/login.html/../caigo → 解析为 /caigo
```

安全影响：可能绕过基于原始路径的安全检查

原因：遵循Servlet规范对路径的标准化处理

#### Spring Boot

Spring Boot 2.3.0 (Spring Framework 5.2.x)

路径解析器：AntPathMatcher 或 PathPatternParser

```java
// 示例：自动规范化路径
String path = "/user/../admin";
path = StringUtils.cleanPath(path); // 结果: "/admin"
```

安全等级：中等（自动规范化但允许路径遍历符号）

  

Spring Boot 2.3.1+ (Spring Framework 5.3.x)

安全改进：spring boot 从2.3.1版本开始默认禁用`../`解析，2.3.0仍可行

```plain
新增匹配策略配置项：
spring.mvc.pathmatch.matching-strategy=ant_path_matcher
spring.mvc.pathmatch.use-trailing-slash-match=true
```

### ;号解析特性

访问路由为：/caigo;/login.html

Tomcat 的 URL 解析机制 会：

1.  将 `;` 视为 参数分隔符（类似 `?` 但非标准）。
2.  默认情况下，`/caigo;/login.html` 会被截断，只保留 `;` 之前的部分，即 `/caigo`。
3.  这种行为可能导致安全问题（如 绕过权限检查）。Tomcat 7.x / 8.x / 9.x / 10.x 均默认启用此行为。

主要影响：如果应用的路由鉴权仅检查 `request.getRequestURI()`，攻击者可能通过 `;` 绕过路径匹配规则。

绕过:

```plain
/test;admin?name=123
```

> Spring boot测试了最新的3.5.0，同理。会解析成/test?name=123

## HTTP请求动词篡改

代码中写了 `if (request.getMethod().equals("GET")) { ... }`，或者 Spring Security 中配置了 `antMatchers(HttpMethod.GET, "/api/admin").hasRole("ADMIN")`。

绕过方式：

将请求包的动词改为 `HEAD`、`OPTIONS`、`POST` 或甚至是不存在的动词（如 `MIST`）。如果业务接口（Controller）恰好使用了包容性极强的 `@RequestMapping(value="/api/admin")`（没限定方法），那么这个非标准动词的请求就会穿透鉴权，直达业务层。

## 代码逻辑缺陷

```java
public void doFilter(...) {
    if (!checkAuth(request)) {
        response.getWriter().write("Unauthorized");
        // 忘记写 return; 了！
    }
    // 鉴权失败，但代码依然会继续往下执行，放行请求！
    chain.doFilter(request, response);
}
```

绕过方式：

攻击者直接发包，虽然收到了 "Unauthorized" 的响应体内容，但实际上业务代码（如删除数据、修改密码）在后端已经被 `chain.doFilter` 成功触发执行了。

## 安全框架特性误用

### Shiro

Shiro 的 `/*` 与 `/**` 混淆：

-   `/*` 只匹配一层路径（如 `/api/login`）。
-   `/**` 匹配所有子路径（如 `/api/admin/delete`）。

绕过方式：

如果配置了 `map.put("/admin/*", "authc")`，攻击者只要访问 `/admin/user/delete`，由于多了一层目录，不在 `/*` 的管辖范围内，直接越权绕过。

### Spring Security

Spring Security 权限覆盖：

在 Spring Security 中，规则是“按顺序匹配，先到先得”。  
如果开发者把 `.antMatchers("/**").permitAll()` 写在了 `.antMatchers("/admin/**").authenticated()` 的前面，那么整个系统就全部裸奔了。

## JWT/Token 校验逻辑缺陷

审计特征： 查找 `Jwts.parser().setSigningKey(key).parseClaimsJws(token)` 之类的代码。

绕过方式：

1.  硬编码密钥： 密钥写死在代码里，攻击者提取后直接伪造超级管理员 Token。
2.  弱校验： 代码里只提取了解析出来的 payload 信息（如 userId），却没有捕获 `SignatureException` 或没校验签名有效性。攻击者修改 Token 中间的载荷，直接提权。
3.  Alg=None 漏洞： 框架较老，未强制禁用 `none` 算法，攻击者把 Header 里的加密算法改为 `none`，去掉签名部分直接绕过校验。
